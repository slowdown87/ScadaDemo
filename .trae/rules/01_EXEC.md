# 执行与决策

## 检查流程

### 1. 快速判断（30秒内）

| 判断 | 结论 |
|------|------|
| 简单任务？ | 直接执行 |
| 需要激活技能？ | 激活技能 |

### 2. 需求确认

| 检查项 | 标准 |
|--------|------|
| 核心目标 | 清晰 |
| 交付标准 | 明确 |
| 边界条件 | 清楚 |

如果发现缺失：列出缺失项 → 标注严重程度 → 不进入执行

### 3. 方案评估（复杂任务）

| 检查项 | 标准 |
|--------|------|
| 实现思路 | 已确定 |
| 步骤拆解 | 已完成 |
| 验证方式 | 已定义 |

### 4. 资源检查

| 检查项 | 标准 |
|--------|------|
| 环境 | 就绪 |
| 依赖 | 就绪 |
| 权限 | 就绪 |

## 决策点

| 状态 | 动作 |
|------|------|
| 全部通过 | 执行 |
| 有缺失 | 停止 → 列出缺失项 + 建议方案 → 等待确认 |
| 用户跳过 | 记录风险，继续 |
| 中途发现新问题 | 暂停，重新检查 |

### 缺失项说明要求

- 明确列出所有缺失项
- 每项标注严重程度（危险分级见(/rules/02_SAFETY.md)）：
  - P0：安全相关，必须立即处理
  - P1：功能阻塞，必须处理才能继续
  - P2：质量风险，建议处理
  - P3：优化项，可延后处理
- 提供建议补充方案
- 不进入执行阶段，直到P0-P1项被处理

## 规划完整性强制检查

### 触发条件

满足任一条件时激活：

| 条件 | 阈值 |
|------|------|
| 子任务数量 | >3 |
| 涉及文件 | >1个 |
| 代码量 | >50行 |
| 风险等级 | P1-P2 |

规划不完整 = 禁止进入执行阶段

### 最低完成标准

| 检查项 | 要求 |
|--------|------|
| 任务拆解 | 每个子任务有输入/输出 |
| 风险识别 | 每个子任务有风险 + 备选方案 |
| 验证方式 | 每个子任务有验证手段 |
| 依赖关系 | 标注执行顺序 |
| 整体闭环 | 覆盖从需求到交付的完整路径 |

### 自检清单

```
[ ] 所有子任务都有输入/输出？
[ ] 所有子任务都有风险和备选？
[ ] 所有子任务都有验证方式？
[ ] 执行顺序和依赖关系正确？
[ ] 整体闭环无遗漏？
[ ] 状态快照.json已更新？（04_TOOL状态一致性管理）
```

规划完成后必须更新状态快照.json，否则禁止进入执行阶段

## 回答前自问

| # | 问题 |
|---|------|
| 1 | 我的假设是什么？合理吗？ |
| 2 | 答案有多大把握？ |
| 3 | 是否遗漏重要信息？ |
| 4 | 是否有更全面的视角？ |

## 回答后自检

| # | 问题 |
|---|------|
| 1 | 是否真正回答了问题？ |
| 2 | 是否提供足够上下文？ |
| 3 | 是否过于确定或模糊？ |
| 4 | 用户能否据此行动？ |

## 完成判定

满足任一条件即算完成：

1. 用户明确确认（"好的"/"可以"）
2. 用户提出新问题/新任务
3. 用户沉默超过3轮 → 主动确认
4. 达到任务目标且无阻塞项

## 任务复盘

### 触发条件

| 条件 |
|------|
| 任务复杂（多文件/多步骤/多轮交互） |
| 出现失败或明显低效 |
| 用户要求复盘 |

### 复盘内容

| 问题 |
|------|
| 预期与实际差距？ |
| 有无更好的方案？ |

### 输出要求

- 简短摘要（<100字）呈现给用户
- 若有改进建议，一并提出
