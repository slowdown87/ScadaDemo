# 多角色模拟技能

在单次回复中模拟多个专业角色协作，兼顾不同视角和专业能力。

## 核心思想

不是让一个AI假装是多个，而是让回答**覆盖完整的决策链**：
- 架构师视角 → 全局最优
- 开发者视角 → 可行性
- 测试视角 → 质量保障
- 审查视角 → 风险控制

## 内置角色

| 角色 | 职责 | 关注点 |
|-----|------|-------|
| **架构师** | 分析问题、设计方案 | 全局、扩展性、技术趋势 |
| **开发者** | 编写代码、实现方案 | 可行性、性能、代码风格 |
| **测试** | 验证质量、发现缺陷 | 边界条件、错误处理、覆盖率 |
| **审查** | 代码审查、风险评估 | 安全、性能瓶颈、最佳实践 |

## 协作模式

### 模式1：线性协作（适合简单任务）

```
架构师 → 开发者 → 测试 → 完成
```

### 模式2：并行协作（适合复杂任务）

```
         → 开发者A →
架构师 →           → 整合 → 完成
         → 开发者B →
```

### 模式3：迭代协作（适合高风险任务）

```
架构师 → 开发者 → 测试
  ↑                    ↓
  ← ← ← ← ← ← ← ← ← ←
       迭代直到通过
```

## 输出格式

```
## [角色名称]

**分析**：
...

**建议/输出**：
...

---

## [下一个角色]

**基于前序角色的输出**：
...

**补充**：
...
```

## 使用场景

| 场景 | 角色组合 |
|-----|---------|
| 新功能开发 | 架构师 + 开发者 + 测试 |
| Bug修复 | 开发者 + 测试 + 审查 |
| 代码审查 | 审查（主导）+ 开发者（确认）|
| 架构决策 | 架构师（主导）+ 开发者（评估）|
| 复杂项目 | 全部角色 |

## 工作流程

```
1. 明确任务和目标受众
2. 确定需要的角色组合
3. 角色依次发言，后一个基于前一个的输出
4. 最终整合为完整方案
5. 如有冲突，明确标注并给出权衡建议
```

## 冲突处理

当角色之间有冲突时：

```
## 冲突：XXX

- **架构师观点**：...
- **开发者观点**：...

**权衡建议**：
1. 如果A优先：...
2. 如果B优先：...

**推荐**：B优先，因为...
```

## 限制

- 模拟角色受限于单个模型能力
- 无法真正并行执行
- 复杂任务建议拆分为多次对话

## 提示词

激活此技能时，可以说：

```
"用多角色模式分析这个需求"
"请架构师、开发者、测试三个角色协作完成这个任务"
"从架构和实现两个角度评估这个方案"
```
