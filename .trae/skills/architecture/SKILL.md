# 架构设计技能

从全局视角设计系统架构，做出明智的技术决策。

## 架构思维框架

### 1. 理解需求
```
- 功能需求：系统要做什么？
- 非功能需求：性能、安全、可扩展性？
- 约束条件：时间、技术、团队？
```

### 2. 方案探索
```
- 至少3个候选方案
- 每个方案的优缺点
- 权衡取舍明确
```

### 3. 决策评估
```
| 维度 | 方案A | 方案B | 方案C |
|-----|------|-------|-------|
| 扩展性 | | | |
| 复杂度 | | | |
| 成本 | | | |
| 风险 | | | |
```

## 常见架构模式

| 模式 | 适用场景 | 优点 | 缺点 |
|-----|---------|-----|------|
| **分层架构** | 一般应用 | 清晰简单 | 灵活性差 |
| **微服务** | 大型复杂系统 | 独立部署 | 复杂度高 |
| **事件驱动** | 实时系统 | 解耦异步 | 调试难 |
| **CQRS** | 高并发读写 | 性能优化 | 学习曲线 |
| **DDD** | 复杂业务 | 业务对齐 | 成本高 |

## 技术选型评估

### 评估维度
```
1. 成熟度：社区活跃度、文档、案例
2. 性能：基准测试、实际场景
3. 成本：许可、基础设施、人力
4. 风险：供应商锁定、技术过时
5. 团队：现有技能、学习曲线
```
