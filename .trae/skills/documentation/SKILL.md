# 文档编写技能

创建完整、准确、易维护的技术文档。

## 文档类型

| 类型 | 受众 | 内容重点 |
|-----|------|---------|
| **README** | 所有开发者 | 项目概览、快速上手 |
| **API文档** | 调用方 | 接口说明、参数、示例 |
| **架构文档** | 架构师/开发者 | 系统设计、技术选型 |
| **开发指南** | 贡献者 | 开发流程、代码规范 |
| **运行手册** | 运维/部署 | 环境配置、部署步骤 |

## 文档编写原则

### 内容的KRUD
```
K - Keep it complete（完整）
R - Keep it relevant（相关）
U - Keep it up-to-date（时效）
D - Keep it distinct（不重复）
```

### 写作要点
```
1. 简洁：不说废话
2. 结构：分层标题
3. 示例：代码示例不可少
4. 更新：随代码同步更新
```

## 文档模板

### README.md
```
# 项目名称

一句话描述

## 快速开始
安装 / 运行步骤

## 功能特性
核心功能列表

## 文档
详细文档链接

## 贡献
如何贡献

## 许可
```

### API文档
```
## 接口名称

**描述**：...

**请求**：
- 方法：GET/POST
- 路径：/api/...
- 参数：...

**响应**：
- 成功：200
- 失败：400/401/500

**示例**：
\`\`\`json
请求示例
\`\`\`
```

### 架构文档
```
## 概述
系统背景和目标

## 架构图
[架构图]

## 核心组件
| 组件 | 职责 | 技术栈 |

## 数据流
数据如何流转

## 部署架构
如何部署
```

## 自检清单

- [ ] 受众明确？
- [ ] 结构清晰？
- [ ] 示例完整？
- [ ] 时效最新？
- [ ] 格式一致？

## 常用位置
```
/README.md           ← 项目入口
/docs/               ← 详细文档
/api/                ← API文档
/ARCHITECTURE.md     ← 架构文档
/CONTRIBUTING.md     ← 贡献指南
```

## 配合使用

- 结合 architecture 编写架构文档
- 结合 code-review 在审查中检查注释
- 结合 tool_rules 创建/更新文件
