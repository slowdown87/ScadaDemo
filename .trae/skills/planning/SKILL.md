# 任务规划技能

你将进入最先进的任务规划模式，使用 ReAct + Task Decomposition + Self-Correction 框架。

## 核心框架

### ReAct 循环

```
Thought → Action → Observation → Thought → ... → Final Answer
```

- **Thought**：分析当前状态，决定下一步行动
- **Action**：执行工具/命令
- **Observation**：收集执行结果
- **循环**：直到任务完成或达到限制

### Task Decomposition

**拆解策略**：
- **Decomposition-First**：稳定环境，先完整规划再执行
- **Interleaved**：动态环境，边规划边执行

**拆解原则**：
- 子任务需语义独立，减少推理错误传播
- 每个子任务有明确输入输出
- 支持并行执行的子任务应分开

### Self-Correction

**失败处理**：
```
失败 → 分析原因 → 调整策略 → 重试或替换方案
```

**反思检查**：
- 我的假设还成立吗？
- 是否需要回退或重新规划？
- 有没有更好的路径？

## 规划流程

```
1. 理解目标
   - 明确最终交付物
   - 识别约束条件（时间、成本、质量）
   - 确定成功标准

2. 任务拆解
   - 分解为原子化子任务
   - 确定依赖关系
   - 识别可并行任务

3. 制定执行计划
   - 排序子任务
   - 预估每步风险
   - 准备备选方案

4. 生成/更新状态快照
   - 根据"状态快照操作规范"更新快照
   - 设置currentPhase为"规划"
   - 设置task.progress为"0/N"
   - 设置task.status为"进行中"

5. 执行与监控
   - 按计划执行
   - 观察每步结果
   - 记录关键决策点
   - 每阶段更新状态快照（progress, currentPhase, blockers）

6. 迭代优化
   - 失败时自我纠错
   - 根据反馈调整
   - 完成时验证结果
```

## 状态快照操作规范

### 状态快照模板

```json
{
  "version": "1.0",
  "created": "<ISO8601格式时间戳>",
  "lastUpdated": "<ISO8601格式时间戳>",
  "currentPhase": "规划",
  "task": {
    "name": "<任务名称>",
    "progress": "0/<子任务总数>",
    "status": "进行中"
  },
  "blockers": [],
  "pendingConfirmations": ["<待确认项>"],
  "lastAgent": "planning"
}
```

### 状态转换规则

| 时机 | currentPhase | task.progress | task.status |
|------|--------------|---------------|-------------|
| 规划开始 | 规划 | 0/N | 进行中 |
| 子任务完成 | 保持 | N/N | 进行中 |
| 执行开始 | 执行 | N/N | 进行中 |
| 任务完成 | 完成 | N/N | 完成 |
| 任务阻塞 | 保持 | N/N | 阻塞 |

### 更新操作

1. 打开 .trae\状态快照.json
2. 按模板创建（如不存在）
3. 按状态转换规则更新字段
4. 保存文件

## 输出格式

### 规划阶段

```
## 执行计划

### 任务拆解
1. [子任务1] - 输入/输出
2. [子任务2] - 输入/输出
3. ...

### 执行顺序
1. → 2. → 3.（顺序）
1. 与 3. 可并行

### 风险点
- 风险1：备选方案
- 风险2：备选方案
```

### 执行阶段

```
## 进度

### 已完成
- [x] 子任务1 - 结果

### 进行中
- 子任务2 - 观察: ...

### 待执行
- 子任务3
```

### 总结阶段

```
## 结果

**完成度**: 100% / 部分完成 / 未完成

**交付物**: ...

**关键决策**: ...

**遗留问题**: ...
```

## 工具使用

配合 tool_rules 使用，常见模式：

```
1. 搜索信息：web search / grep
2. 分析代码：cat / grep / tree
3. 执行命令：python / bash / pytest
4. 验证结果：diff / test / assert
```

---

## 自我检查点

执行每个子任务后问自己：

```
1. 结果符合预期吗？
2. 需要调整后续计划吗？
3. 可以继续还是需要回退？
4. 是否达到成功标准？
```

---

## 适用场景

- 复杂项目开发
- 多步骤调试任务
- 系统架构设计
- 技术选型决策
- 任何需要规划的长任务

---

## 限制

- 单次规划步数有限（通常 10-50 步）
- 复杂任务可能需要多次规划迭代
- 长期任务需要记录中间状态
